# 名古屋飲食店検索・予約サービス「Nagoyameshi」

名古屋への旅行客が「迷わずに最適な食事処を見つけられること」をコンセプトにした、店舗検索・レビュー・予約・決済機能を持つフルスタックWebアプリケーションです。

プログラミングスクールの教材をベースに、実際のプロダクト運用や実務で求められるセキュリティ・管理機能を独自に想定し、大幅な機能拡張を行いました。

---

## 💡 開発の背景と目的（なぜ作ったか）

* **開発のきっかけ：**
  SAMURAI ENGINEERの学習教材をベースに基礎機能を実装。
* **解決したかった課題：**
  数ある飲食店検索サイトの中で、ターゲットを「名古屋への旅行客」に特化。見知らぬ土地でも迷わずに、地元の魅力的な食事処を見つけ、予約までスムーズに完結できる体験を目指しました。
* **実務を意識したアプローチ：**
  「掲載店舗数やユーザー数が増加した未来」を想定し、一般的な学習用アプリでは省略されがちな**セキュリティ対策**や**管理者向けの運用保守機能（CSV出力、ログ出力など）**を自発的に設計・実装しました。

## 🌟 特にこだわった・工夫したポイント

### 1. 堅牢なセキュリティ設計
不特定多数が利用するWebサイトにおいて、不正アクセスからユーザーを守るため、Spring Securityを活用して「5回以上のログイン失敗でアカウントをロックする」仕様を組み込みました。実務レベルのセキュリティ意識を持って実装しました。

### 2. 運用の自動化と効率化（管理者機能の拡充）
サービスの成長に伴い管理者の運用コストが増大することを見越し、手動管理ではなく「予約データのCSV一括出力」や、バグ発生時に迅速に原因究明ができるよう「適切なログファイル出力」を実装。開発者視点だけでなく、**「運用者視点」**を持った設計を意識しました。

## 🚀 開発における困難と学び（💡一番のアピールポイント）

### 【課題】データベース改修時におけるデータ整合性の維持
開発途中、機能拡張に伴いデータベース（MySQL）のスキーマ変更が必要になりました。しかし、テーブル間にリレーション（外部キー制約など）が設定されていたため、そのまま変更を加えようとすると整合性エラーが発生し、開発がストップする課題に直面しました。

### 【解決へのアプローチ】
エラーログを精査し、ただ闇雲にコードを書き換えるのではなく「テーブル間の依存関係」が原因であると仮説を立てました。AIのデバッグアシスタントも活用しながら、「一度テーブル間の関係（制約）を適切に解除してから操作を行う、または移行手順の順番を正しく制御する」必要があることに気づきました。

### 【学び】
この経験を通じて、RDBの設計におけるデータの整合性と制約の重要性を深く理解しました。以降は、安易にDBを修正するのではなく、変更が他のテーブルに与える影響を事前に依存関係図（ER図）などで整理してから作業に移るという、計画的な開発プロセスが身につきました。
